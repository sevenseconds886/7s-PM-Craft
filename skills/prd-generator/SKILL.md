# PRD Generator

将用户的自然语言需求描述，转化为符合 7s-PM-Craft 文件协议的标准需求文档。

## 触发方式

由 workflow-orchestrator 调用，或独立使用。

## 输入

- `description`（string，必填）：用户对产品需求的自然语言描述
- `options`（object，可选）：
  - `title`：需求标题（默认从 description 提取）
  - `priority`：P0 / P1 / P2 / P3 / P4 / P5（默认 P2）
  - `platform`：["web"] / ["mobile"] / ["web", "mobile"]（默认 ["web"]）
  - `product_line`：产品线名称（默认"默认产品线"）
  - `sprint`：所属迭代（默认空）
  - `developer`：开发负责人（默认空）
  - `requester`：需求提出人（默认空）
  - `due_date`：预计上线日期 YYYY-MM-DD（默认空）
  - `tags`：标签数组（默认空）

## 输出

一个完整的 `requirement.md` 文件内容（字符串），包含：

1. **YAML Front Matter**（元数据区）
2. **需求描述**（用户描述的扩展和结构化）
3. **验收标准**（可勾选的清单）
4. **备注**（额外说明）

## 生成规则

### 1. YAML Front Matter 必须完整

```yaml
---
id: REQ-000001          # 由 orchestrator 分配，此处保留占位
title: 登录页面
status: 设计中
priority: P0
platform:
  - web
  - mobile
product_line: 产品线A
sprint: 迭代1
developer: ""
requester: ""
created: 2026-04-30
updated: 2026-04-30
due_date: ""
tags:
  - 认证
  - 核心流程
---
```

**规则：**
- `id`、`created`、`updated` 由 orchestrator 自动填充，生成时保留 `""` 或占位符
- `status` 固定为"设计中"
- 所有字段必须有值，空值用 `""` 或 `[]` 表示，不能省略字段
- `platform` 必须为数组格式
- `tags` 必须为数组格式

### 2. 需求描述要结构化

将用户的自然语言描述扩展为结构化的需求说明：

```markdown
## 需求描述

### 背景
[简述这个需求的产品背景和目标]

### 功能说明
[分点说明具体功能]

1. **功能点A**：...
2. **功能点B**：...

### 用户旅程
[描述用户如何一步步使用这个功能]

1. 用户打开...
2. 用户点击...
3. 系统响应...
```

### 3. 验收标准要可测试

根据需求自动生成可验证的验收标准：

```markdown
## 验收标准

- [ ] 标准1：[具体、可测试的描述]
- [ ] 标准2：[具体、可测试的描述]
- [ ] 标准3：[具体、可测试的描述]
```

**规则：**
- 每条标准必须以 `- [ ] ` 开头
- 描述要具体、可验证，避免模糊词汇（如"好用""美观"）
- 至少生成 3-5 条标准
- 涉及前端交互的，要说明触发条件和预期结果

### 4. 备注区留空或补充

```markdown
## 备注

[补充任何需要特别注意的事项，如技术约束、依赖关系、参考链接等]
```

## 示例

**输入：**
```
description: "做一个登录页面，用户用手机号+验证码登录，60秒倒计时，登录成功后跳到首页"
options:
  priority: P0
  platform: ["web", "mobile"]
  product_line: "用户中心"
  sprint: "迭代1"
```

**输出（requirement.md）：**

```markdown
---
id: ""
title: 手机号验证码登录
status: 设计中
priority: P0
platform:
  - web
  - mobile
product_line: 用户中心
sprint: 迭代1
developer: ""
requester: ""
created: ""
updated: ""
due_date: ""
tags:
  - 认证
  - 登录
---

## 需求描述

### 背景
为了提升用户体验并降低密码管理成本，系统需要支持手机号 + 短信验证码的登录方式。

### 功能说明

1. **手机号输入**：用户输入 11 位中国大陆手机号，实时校验格式
2. **验证码发送**：点击"获取验证码"后，后端发送 6 位数字验证码，前端开始 60 秒倒计时
3. **验证码输入**：用户输入收到的验证码，支持一键填充
4. **登录确认**：点击"登录"后验证手机号与验证码匹配，成功后跳转首页
5. **错误处理**：手机号格式错误、验证码错误、验证码过期均给出明确提示

### 用户旅程

1. 用户访问登录页，看到手机号输入框
2. 输入手机号，格式正确后"获取验证码"按钮变为可点击
3. 点击按钮，开始 60 秒倒计时（期间按钮禁用并显示剩余秒数）
4. 用户收到短信验证码，输入到验证码框
5. 点击"登录"，验证通过后自动跳转首页

## 验收标准

- [ ] 手机号输入框仅允许输入数字，且最多 11 位
- [ ] 输入非 11 位手机号时，"获取验证码"按钮保持禁用状态
- [ ] 点击"获取验证码"后，按钮禁用并开始 60 秒倒计时，显示"已发送（59s）"
- [ ] 倒计时结束后，按钮恢复为"重新获取"
- [ ] 输入正确手机号和有效验证码后点击登录，3 秒内跳转首页
- [ ] 验证码错误或过期时，显示明确错误提示（如"验证码错误，请重新输入"）
- [ ] 登录状态在页面刷新后保持（使用 localStorage 或 cookie）

## 备注

- 短信验证码有效期为 5 分钟
- 同一手机号 1 分钟内不可重复发送验证码
- 参考设计：Ant Design Mobile 登录页
```

## 质量检查清单

生成完成后，请自检：

- [ ] YAML Front Matter 所有字段都存在
- [ ] `platform` 和 `tags` 是数组格式
- [ ] 需求描述至少包含"背景"和"功能说明"
- [ ] 验收标准不少于 3 条，且都可测试验证
- [ ] 没有使用模糊词汇（好用、美观、快速等）
- [ ] 输出是完整的 Markdown 字符串，可直接写入 requirement.md
