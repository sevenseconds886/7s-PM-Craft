> 状态：已确认，可执行

---

## 1. 项目概述

将产品日常工作流与 AI 生产力工具深度嵌合，实现"一句话生成高保真原型 + 自动归档需求文档 + 浏览器端统一管理"的闭环系统。

**核心价值：**
- 减少产品从需求构思到原型交付的重复劳动
- 用文件协议替代数据库，实现真正的开箱即用和跨设备迁移
- 浏览器端同屏展示原型与需求，提升与开发沟通效率

---

## 2. 目标用户与场景

**用户画像：** 产品经理（非研发背景），个人使用为主，未来可能分享给团队成员查看。

**典型场景：**

| 场景 | 当前痛点 | 系统解决方式 |
|------|----------|-------------|
| 写需求文档 | 重复劳动，格式不统一 | 输入自然语言，AI 自动生成规范需求.md |
| 画原型 | 工具切换成本高，低保真难沟通 | AI 直接输出高保真 HTML 原型 |
| 需求管理 | 散落在各处，状态靠记忆 | 浏览器端统一管理，状态可视化 |
| 与开发沟通 | 原型和需求分开看，来回切换 | 同屏左原型右需求，一目了然 |
| 换电脑 / 分享 | 数据绑死在某台电脑或某个工具 | 纯文件驱动，拖文件夹即走 |

## 3. 系统架构

系统采用三层架构，自上而下：

```
┌─────────────────────────────────────┐
│  浏览器平台层（Web 界面）            │  ← 需求聚合、展示、管理
│  · 首页（产品线 + 统计 + 设置）      │
│  · 需求列表（状态视图 + 迭代视图）   │
│  · 需求详情（同屏原型 + 需求文档）   │
├─────────────────────────────────────┤
│  Skills 层（IDE / WorkBuddy 调用）  │  ← 生产内容
│  · prd-generator（生成需求文档）    │
│  · prototype-generator（生成原型）  │
│  · workflow-orchestrator（编排命令）│
├─────────────────────────────────────┤
│  文件协议层（本地文件夹）            │  ← 数据存储
│  · PROTOCOL.md（显式规则文件）      │
│  · 目录结构规范                     │
│  · 命名规则                         │
│  · 状态流转规则                     │
│  · 归档机制                         │
└─────────────────────────────────────┘
```

**设计原则：文件即数据库。** 所有数据保存在本地文件系统中，不依赖外部数据库。

## 4. 文件协议层（详细规范）

### 4.1 目录结构

```
📁 {workspace}/
├── 📁 products/                          # 活跃需求
│   └── 📁 {product-line}/                # 产品线
│       └── 📁 REQ-{id}-{slug}/           # 单个需求文件夹
│           ├── requirement.md
│           ├── prototype-web.html        # 可选
│           └── prototype-mobile.html     # 可选
├── 📁 archive/                           # 归档需求
│   └── 📁 {product-line}/
│       └── 📁 REQ-{id}-{slug}/
│           ├── requirement.md
│           ├── prototype-web.html
│           └── prototype-mobile.html
├── 📄 PROTOCOL.md                        # 文件协议规则（显式定义）
├── 📄 .workflow-config.json              # 可选：工作区配置
└── 📄 .sprints.json                      # 迭代元数据（轻量JSON，不污染需求文档）
```

**规则说明：**
- `{id}`：需求编号，6 位数字，全局自增，如 `000001`、`000042`
- `{slug}`：需求简称，英文或拼音，短横线连接，如 `login-page`、`user-center`
- 文件夹名即唯一标识：`REQ-000001-login-page`
- 所有元数据从 `requirement.md` 的 YAML Front Matter 读取，不依赖文件夹名解析
- **平台属性移至元数据**，不再作为文件夹层级
- 多端需求在同一文件夹内放置 `prototype-web.html` 和 `prototype-mobile.html`

### 4.2 requirement.md 格式规范

每个需求文档采用 Markdown 格式，顶部包含 YAML Front Matter 元数据：

```markdown
---
id: REQ-000001
title: 登录页面
status: 设计中
priority: P0
platform:
  - web
  - mobile
product_line: 产品线A
sprint: 迭代1
developer: 张三
requester: 李四
created: 2026-04-29
updated: 2026-04-29
due_date: 2026-05-15
tags:
  - 认证
  - 核心流程
---

## 需求描述
用户可以通过手机号 + 验证码登录系统。

## 验收标准
- [ ] 输入框支持 11 位手机号校验
- [ ] 验证码 60 秒倒计时
- [ ] 登录成功后跳转首页

## 备注
原型包含两种状态：未填写 / 验证中。
```

**字段说明：**

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| id | 是 | string | 需求唯一编号，如 REQ-000001 |
| title | 是 | string | 需求标题 |
| status | 是 | enum | 见下方状态枚举 |
| priority | 否 | enum | P0 / P1 / P2 / P3 / P4 / P5，默认 P2 |
| platform | 否 | array | web / mobile，可多选 |
| product_line | 否 | string | 所属产品线 |
| sprint | 否 | string | 所属迭代名称，空表示未分配。迭代信息存储在 `.sprints.json` |
| developer | 否 | string | 开发负责人 |
| requester | 否 | string | 需求提出人 |
| created | 否 | date | 创建日期 YYYY-MM-DD |
| updated | 否 | date | 最后更新日期 |
| due_date | 否 | date | 预计上线日期 |
| tags | 否 | array | 标签列表 |

**状态枚举（status）：**

```
设计中 → 待评审 → 开发中 → 待验收 → 归档
           ↓
         挂起（可从任何状态进入，可恢复）
```

- **设计中**：需求正在编写/原型制作中
- **待评审**：需求已完成，等待评审
- **开发中**：开发正在进行
- **待验收**：开发完成，等待验收
- **归档**：需求已完成或废弃，归档
- **挂起**：需求暂停，可恢复到之前状态

浏览器端修改状态后，`updated` 字段自动更新。

**优先级枚举：**

| 优先级 | 说明 |
|--------|------|
| P0 | 最高优先级，必须立即处理 |
| P1 | 高优先级，当前迭代必须完成 |
| P2 | 中优先级，建议当前迭代完成 |
| P3 | 低优先级，可排入后续迭代 |
| P4 | 很低优先级，有空再做 |
| P5 | 最低优先级，仅记录 |

### 4.3 归档机制

- 归档操作由浏览器端触发
- 管家执行：将需求文件夹从 `products/{product-line}/` 移动到 `archive/{product-line}/`
- 移动后 `requirement.md` 中的 `status` 自动更新为 `归档`
- 归档需求在浏览器中默认隐藏，可通过筛选器查看

### 4.4 原型文件规范

- 文件名：`prototype-web.html`、`prototype-mobile.html`
- 使用 Tailwind CSS 实现高保真样式（通过 CDN 引入）
- 支持单页面或多页面（多页面时在同一文件夹内命名 `prototype-web-p2.html` 等）
- 部分需求可省略原型文件（由 Skills 控制 `--skip-prototype`）
- 支持原型复用：A 需求引用 B 需求的原型文件（通过 `prototype_ref` 字段标记）

### 4.5 迭代元数据存储（.sprints.json）

迭代信息单独存储在项目根目录的 `.sprints.json` 中，不混入需求文档：

```json
{
  "sprints": [
    {
      "name": "迭代1",
      "status": "active",
      "created": "2026-04-01",
      "closed": null
    },
    {
      "name": "迭代2",
      "status": "active",
      "created": "2026-04-15",
      "closed": null
    },
    {
      "name": "迭代0",
      "status": "closed",
      "created": "2026-03-01",
      "closed": "2026-03-31"
    }
  ]
}
```

**规则：**
- 迭代没有独立状态，只有 `active`（开启）和 `closed`（关闭）
- 需求通过 `sprint` 字段关联迭代名称
- 关闭迭代时更新 `closed` 字段，已关闭迭代默认折叠展示
- 轻量 JSON 格式，人可读，机器易解析

### 4.6 PROTOCOL.md 规则文件

项目根目录放置 `PROTOCOL.md`，显式定义文件协议：

```markdown
# 产品工作流文件协议

## 目录结构
- products/{product-line}/REQ-{id}-{slug}/
- archive/{product-line}/REQ-{id}-{slug}/

## 文件命名
- requirement.md
- prototype-web.html
- prototype-mobile.html

## 元数据字段
（见 requirement.md 格式规范）

## 读取容错
- 缺失 status → 默认"设计中"
- 缺失 priority → 默认"P2"
- 缺失 platform → 默认["web"]
- 缺失 sprint → 默认空字符串
```

任何 Skills（自有或第三方）都遵循此规则输出文件，管家按此规则读取和解析。

## 5. Skills 层

### 5.1 Skills 拆分设计

按职责拆分为三个独立 Skills，通过文件系统关联：

| Skill | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **prd-generator** | 生成需求文档 | 自然语言描述 | requirement.md |
| **prototype-generator** | 生成高保真原型 | 自然语言描述 / requirement.md | prototype-web.html / prototype-mobile.html |
| **workflow-orchestrator** | 命令编排、关联、文件管理 | 用户命令 + PROTOCOL.md | 按规则写入文件系统 |

**关联方式：**
- `workflow-orchestrator` 是入口 Skill，解析用户命令
- 当用户输入 `/prd-new` 时，orchestrator 调用 `prd-generator` 生成文档
- 当用户输入 `/prd-new`（无 `--skip-prototype`）时，orchestrator 再调用 `prototype-generator` 生成原型
- 各 Skill 之间通过**文件系统**关联：一个 Skill 的输出文件是下一个 Skill 的输入
- 第三方可以只替换其中一个 Skill（比如用更好的原型生成器），不影响整体工作流

### 5.2 命令设计（开放协议）

由 `workflow-orchestrator` Skill 统一暴露命令，各平台（WorkBuddy / Cursor / Trae 等）自行解析：

```yaml
commands:
  - name: prd-new
    description: 创建新需求
    args:
      - name: description
        required: true
    options:
      - name: doc-only
        description: 仅生成需求文档（调用 prd-generator）
      - name: skip-prototype
        description: 跳过原型生成
      - name: platform
        description: 指定平台（web/mobile/both）
    workflow: |
      1. 读取 PROTOCOL.md 规则
      2. 分配下一个可用 ID（全局6位自增）
      3. 调用 prd-generator 生成 requirement.md
      4. 如无 --skip-prototype，调用 prototype-generator 生成 prototype
      5. 按规则写入文件系统

  - name: prd-update
    description: 更新已有需求
    args:
      - name: id
        required: true
      - name: description
        required: true

  - name: prd-status
    description: 修改需求状态
    args:
      - name: id
        required: true
      - name: status
        required: true

  - name: prd-archive
    description: 归档需求
    args:
      - name: id
        required: true
```

### 5.3 原型质量要求

- **高保真**：包含真实配色、间距、圆角、阴影
- **可交互**：按钮有 hover 状态，表单可输入（演示级）
- **响应式**：如指定 platform 为 mobile，生成移动端适配布局
- **自包含**：单个 HTML 文件，所有样式内联或 CDN 引入

### 6.3 全局交互规则

| 规则 | 说明 |
|------|------|
| 迭代规则 | 迭代没有状态属性；需求不可同属于多个迭代 |
| 视图规则 | 状态视图 = Tag 页分类列表；迭代视图 = 分段列表；两种视图均支持拖动 |
| 行内编辑 | 支持优先级下拉、状态下拉、迭代下拉 |
| 拖动操作 | 状态视图下拖动切换状态；迭代视图下拖动切换迭代 |

### 6.4 浏览器与本地文件联动机制

由于浏览器安全限制无法直接读写本地文件，采用**本地 HTTP 管家**作为中间层：

```
浏览器 ←──HTTP API──→ 本地管家（Node.js） ←──文件IO──→ 本地文件夹
```

**管家提供的 API：**

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/requirements` | GET | 扫描并返回所有需求列表 |
| `/api/requirements/:id` | GET | 返回单个需求的完整数据 |
| `/api/requirements/:id/status` | POST | 修改需求状态 |
| `/api/requirements/:id/sprint` | POST | 修改需求所属迭代 |
| `/api/requirements/:id/priority` | POST | 修改需求优先级 |
| `/api/requirements/:id/archive` | POST | 归档需求（移动文件夹） |
| `/api/requirements/:id/content` | POST | 修改需求文档内容 |
| `/api/sprints` | GET | 获取所有迭代列表 |
| `/api/sprints` | POST | 创建新迭代 |
| `/api/sprints/:name/close` | POST | 关闭迭代 |

**扫描策略：** 启动时全量扫描 + 浏览器端手动刷新按钮。无需实时文件监听，降低复杂度。

## 7. 开箱即用设计

### 7.1 零配置原则

- 无外部数据库依赖
- 无环境变量配置（所有配置从文件协议推断）
- 无绝对路径（全部基于工作区相对路径）

### 7.2 换电脑迁移流程

```
1. 新电脑安装 Node.js（官网下载安装包，或携带便携版）
2. 将整个工作区文件夹拖入新电脑
3. 在工作区根目录执行：npm start
   → 自动安装依赖 → 启动管家 → 输出访问地址
4. 浏览器打开 http://localhost:3000
5. 完成，所有历史需求自动加载
```

### 7.3 自动识别机制

- 管家通过**扫描文件系统**发现需求，而非查询数据库
- 识别条件：文件夹名匹配 `REQ-*` 模式 + 内部存在 `requirement.md`
- 用户手动创建、修改、重命名文件夹后，刷新浏览器即可同步
- 修改 `requirement.md` 中的元数据后，刷新即可更新

## 8. 技术选型建议

| 层级 | 推荐方案 | 理由 |
|------|----------|------|
| 本地管家 | Node.js + Express | 轻量、用户已安装、生态丰富 |
| 前端界面 | Vue 3 + Vite 或纯 HTML/JS | 开发快，打包后无依赖 |
| 原型样式 | Tailwind CSS (CDN) | AI 生成时直接写类名，开箱即高保真 |
| Skills | 开放协议，多平台适配 | 不绑定特定平台，未来可开源 |
| 文件协议 | Markdown + YAML Front Matter | 人可读，机器易解析 |
